동기와 비동기에 대한 간단한 이해

1. 자바스크립트의 구성 요소

  • stack


    자바스크립트 코드가 실행되면 실행될 명령들(스택 프레임)이 쌓이는 장소.
    스텍에 새로운 프레임들이 들어오고 실행이 완료되면 나간다.
    드럼통 처러 새겨서 First In Last Out 구조의 자료형이다. (먼저 들어온놈이 나중에 나가는 구도...)

  • Heep


    동적으로 생성된 변수들을 메모리에 heep에 가서 저장이 된다.
    저장공간

  • 데스크 큐(Queue)

    웹 api로 부터 받은 데스크를 큐에 저장해 둔다. 스텍이 비어지면 이벤트 루프가 큐에 아이템을 꺼내다가 스텍으로 올려준다.
    First In First Out의 자료구조 형태이다

  • Event Loop


    스텍과 큐 사이에서 흐르을 제어한다. 이벤트 루프는 스텍이 비어있는지를 확인하고, 비어있으면 큐에 있는 아이템을 꺼내다가 스텍에 올려준다.

  • 웹 api


    Ajax 요청, setTimeout(), 이벤트 헨들러의 등록과 같이 웹 브라우저에서 제공하는 기능들을 말한다
    이들은 자바스크립트의 쓰레드(스텍)에서 작업이 이루어지는 것이 아닌, 별도의 쓰레드에서 이루어진다.

  • setTimeout(함수, 시간)


    Ajax, Axios, fetch : 클라이언트와 서버간에 데이터를 주고받는 기술
    Event Handler : 클릭과 같은 이벤트를 핸들링 하는 함수

  • 쓰레드(Thread)


    쓰레드란 실제로 작업을 실행하는 주체이다
    쓰레드는 1개이거나 여러개일 수 있다.
    자바스크립트는 싱글(1개)의 쓰레드만 있다.
    쓰레드가 1개이면 하나씩 밖에 일을 처리 못해서 동기적으로 일을 처리하게 된다.
    멀티 쓰레드라면 (1개 이상) 여러개의 작업을 병렬적으로 처리할 수 있다.
    자바스크립트는 싱글쓰레드를 "동기적 처리"를 한다는 것을 필히 명심해야 된다.

  • API를 부르는 방법

    ajax, axios, fetch가 있다.
    이 중에는 fetch가 제일 간단하다.
    ajax, axios는 React.js에서 가장 많이 사용되는 방식이기도 하다.

  • Promise

    자바스크립트에서 비동기 동장을 다루는 ES6 최신 문법이다.
    기존에 콜백함수를 이용해서 비동기처리를 하였는데,
    $.ajax("http://api.newscatcher.com/v2/search", (result) =>{console.log(result);});
    위의 코드와 같은 방식이라면 url이 호출이 되고, 결과가 나오면 두번째 매개변수인 콜백함수를 불러서
    result에 결과값을 넘겨줬었다. 이렇게 하면 가독성이 좀 많이 떨어진다.(잘 알고 계시는 콜백함수...)
    함수안에 함수, 그 안에 또 함수, 또 함수, 함수 함수 함수... 콜백지옥이란 이런것이다.
    그래서 나온 게 Promise 다.
    function delayP (sec) {
       return new Promise ((resolve, reject) => {
         setTimeout(() => {
          resolve ("success")
        }), sec * (1000)
      })
    } new Promise를 이용해서 promise를 생성하고, promise 안에서는 resolve와 reject 함수를 자동으로 가진다
    resolve는 결과가 문제없이 성공적으로 수행될 때 호출하는 함수
    reject는 중간에 에러가나면 에러를 처리할 수 있는 함수이다.
    resolve가 호풀이 되면 악마적인 콜백 대신에 .then()이나 async / await를 이용해서 그 결과값을 받아내서 다른 일들을 할 수 있다.

  • async / await

    async / await는 동기적인 자바스크립트를 비동기적으로 처리하기 위해서 쓴다.
    fetch는 promise를 리턴한다. 그리고 우리가 필요한건 그 promise 속에서 api호출이 잘 이루어져
    데이터를 받은 다음, promise가 성공적으로 resolve 호출하기를 기다리는 것이다.
    이걸 해주는 게 await 이다.
    await이 없으면 그걸 날 것 그대로의 promise가 리턴이 된다.
    await을 쓰려면 함수를 async로 선언해줘야 한다, 즉 둘을 세트라고 생각하면 된다.
    그 외에도 async로 함수를 선언하면, 그 함수는 자동으로 promise를 방환한다.
    위의 말의 뜻은 또 이 함수를 받아다가 다른 비동기 작업이 가능하다는 이야기가 된다

  • json

    서버 클라이언트 통신에서 많이 쓰이는 데이터 타입 (phg, jpg 뭐 이런 거랑 같이 그냥 데이터 타입이다.)
    객체랑 똑같이 생긴 텍스트라고 이해하면 좋다
    간단한 텍스트인데 객체랑 똑같아서 나중에 읽어오기도 매우 편하다.
    그래서 json타입을 서버통신시에 많이 사용한다.

  • 호이스팅 (재방송)

    function과 var로 선언된 함수와 변수는 호이스팅이 된다. 메모리 힙에 미리 저장이 되어있게 된다.
    따라서 순서와 상관없이 아무데서나 부르면 호출이 가능했었지만.. 일반 상식적으로 말이 안되는 거긴 하다.
    그래서 이를 문제 삼아서 나온게 let, const이다. 선언시 호이스팅이 되어서 메모리 힙에 가 있기는 하지만,
    정의가 된 부분에 도달하기 전까지는 TDZ(일시적 사각지대, Temporal Dead Zone)를 만들어서 접근을 제한한다.
    화살표 함수도 let, const를 사용하여 정의한 함수이기 때문에 TDZ의 적용을 받게 된다.

    getNewsByKeyword 관련 내용 ..

    지난번에 addEventListener("click", (event) => getNewsByTopic(event))는 왜 된 건가여?
    지금 이야기 하는 getNewsByKeyword는 토픽과는 다르게 본인이 바로 불러진게 아니라 익명의 함수 안에 들어가있는 모양이기 때문에,
    즉, addEventListener("click", (event) => getNewsByTopic(event))를 실행할 때에는, 함수가 바로 쓰여진 게 아니라, 익명의 함수를 쓴거기 때문에 에러가 나지 않았다.
    그리고 클릭할 때에는 이미 코드를 한 번 쫙! 처음부터 끝까지 실행을 돌린 상태이기 때문에, 토픽에 접근이 가능한 거였다.
    addEventListener("click", (event) => getNewsByTopic(event))의 경우에는 이부분의 코드 실행시 getNewsByKeyword가 직접적으로 호출이 되었기 때문에,
    함수 정의부분 보다 위쪽에 있다면 접근이 불가능 해진 것..

    요약하자면,

    화살표 함수는 정의 전에 호출 할 수 없다... 순서가 매우 중요하다.

  • 코드 리팩토링

    코드를 짜면 가장 중요한건 일단 작동되는 게 중요하지만,
    이왕이면 좀 더 깔끔하고, 세련되게 코드를 짜는 게 좋다고 할 수 있다. 그래야지 코드의 유지보수, 확장이 쉽다.
    코드 작성의 기본원칙 두 가지, 코드의 안정성과 코드의 확장성 확보!

    코드 리팩토링이 필요한 경우 : 중복되는 코드, 긴 함수 등등이 있다.
    함수는 한가지 일만 하도록 짜여져 있는 것이 좋고, 3번 이상 중복이 되거나, 다른곳에서 많이 쓰여지는 코드는 묶어서 하나의 함수로 하는 것이 좋다.

    }

    console.log("프론트반 가전에 와주셔서 감사하고 항상 건강하세요!")
    }
    const buyCellphone = () => {
    console.log("TV 구매 감사드립니다.")
    console.log("프론트반 가전에 와주셔서 감사하고 항상 건강하세요!")
    }

    어느날, 프론트반 가전회사가 망하고 .. 다시 쥬니어개발자 가전 으로 회사의 이름을 바꾸게 되었다.. 그럼 모든 코드에 프론트반을 쥬니어개발자로 바꾸어야 하는데, 지금 눈에 보이는 제품은 3개가 끝인데, 만약 확장이 잘 되서 100개, 10억개의 제품군이 생겨난다면?
    그리고 다시 회사의 이름이 시니어개발자 가전으로 바꾸어진다면? 그럼 10억군데가 넘는 이름을 하나하나 바꾸어야 할까? 실수라도 오타가 난다면?
    상상만해도 끔찍한 상황이 아닐 수 없다. 이런 코드는 유지보수가 매우매우 힘들고, 무엇인가 더 확장하기도 어렵다. 그래서 우리는 코드를 리팩토링해야 한다.

    const greedMessage = () => {
        console.log("프론트반 가전에 와주셔서 감사하고 항상 건강하세요!") }

    }
    const buyAC = () => {
        console.log("TV 구매 감사드립니다.")
        greedMessage()
    }
    const buyCellphone = () => {

    위와 같은 식으로 코드를 바꾸면, 회사이름이 바뀌어도 greedMessage() 딱 한군데만 코드를 바꿔주면 된다.
    또한 제품군이 늘어나더라도, 간단하게 greedMessage() 만 추가해주면 모둔 제품군이 오타없이 메세지를 보여줄 수 있게 된다.

  • try / catch

    개발자의 기초 소양 중에 하나이다.. 미리 에러가 나올만한 상황을 예측해서 에러핸들링을 적절하게 해주는 것이 유저에 대한 예의...
    try {
        에러가 발생될 수 있는 코드를 입력 ..
    } catch(error) {
        에러 핸들링
    }
    1. try문 안에 있는 코드를 실행하다가 에러가 나면
    2. catch 블럭 안에 들어가서 에러를 실행되면
    3. 에러가 없다면 catch 블럭을 무시하고 넘어간다
    4. try / catch는 오직 런타임 에러에만 동작한다. 즉, 실행가능한 코드에만 try / catch가 사용될 수 있다.
    5. 만약에 {} 중괄호 짝이 안맞다던지, 등등의 문법적인 에러로 돌아가지 않는 코드라면 try / catch가 동작 할 수 없다.

  • 에러 객체 Error

    try에서 에러가 생기면 catch 블럭에 error 정보를 error 객체 안에 넣어서 매개변수로 전달해 준다.
    Error 객체를 살펴보면 안에 name, message 등 여러 이용가능한 속성값이 있지만, 에러를 보기위해 Error.message 만 기억해도 좋다.

  • API 응답에서 에러가 났는 지 확인하는 방법

    1. response.status 를 통해서 아는 방법 : 주로 디테일 하게 어떤 종류인지 에러 코드를 보여준다. (400,401 등등)
    2. response.ok 를 보는 방법 : ok가 true면 정상, false면 비정상, 자세한 코드는 나타나지 않는다.
    3. 응답에 따라서 throw를 이용해서 보여주고 싶은 메세지와 함께 에러를 catch로 던져 줄 수 있다.
    4. 그 외에 내가 에러를 발생시키고 싶은 상황이 있다면, throw를 통해서 강제로 에러를 던져 줄 수 있다.
